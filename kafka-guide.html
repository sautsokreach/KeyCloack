Key difference from RabbitMQ: Kafka retries use separate retry topics (user-events-0, user-events-1) so the main topic is never paused or blocked. RabbitMQ nacks the message and puts it back in the same queue, which can stall other messages behind it if retries keep failing.

Offset commit: if the consumer crashes mid-processing, Kafka replays from the last committed offset. RabbitMQ uses an ack/nack model instead.

Phase 1 — Start the stack

1
Start Kafka (if not already running)
Kafka runs in KRaft mode (no Zookeeper). It will auto-create the topics declared in KafkaConfig.java when the Spring app connects.
# from the project root docker compose up -d kafka # confirm it's healthy docker logs kafka --tail 5
2
Start the Spring Boot app
Watch the startup logs — Spring Kafka will create the user-events topic (3 partitions) and user-events.DLT topic, then the consumer will join the group.
mvn spring-boot:run # expected log lines on startup: INFO AdminClientConfig - Creating topic user-events (partitions=3) INFO KafkaListenerEndpointRegistry - Starting Consumer group=demo-group INFO AbstractCoordinator - [demo-group] Joined group, generation 1

Phase 3 — Basic publish & consume

4
GET /api/kafka/info — understand the setup
Before sending any messages, read the info endpoint. It returns a live cheat-sheet of Kafka concepts and your current config.
curl -s http://localhost:8080/api/kafka/info \ -H "Authorization: Bearer $TOKEN" | python3 -m json.tool
Response: { "topic": "user-events", "partitions": 3, "consumerGroup": "demo-group", "concept": { "key-routing": "same key → same partition → ordered delivery per user", "offset": "position in partition log — consumers track where they left off", "vs-rabbitmq": "Kafka retains messages after consumption (replay possible)" } }
5
POST /api/kafka/send — publish your first message
The message key is your username. This determines which partition receives the message. Watch the app logs to see the partition number and offset printed by the consumer.
curl -s -X POST http://localhost:8080/api/kafka/send \ -H "Authorization: Bearer $TOKEN" \ -H "Content-Type: application/json" \ -d '{"action": "login"}' | python3 -m json.tool
--- Publisher log --- INFO KafkaMessagePublisher - Kafka event published → topic=user-events partition=1 offset=0 key=alice --- Consumer log (few ms later) --- INFO KafkaMessageConsumer - Kafka received → topic=user-events partition=1 offset=0 key=alice INFO KafkaMessageConsumer - event: user=alice action=login INFO KafkaMessageConsumer - Processed successfully.
What to notice: the partition and offset values in the log. Send the same request again — the partition will be the same (same key = same partition), and the offset increments by 1 each time.
6
Send several messages and watch offsets grow
Run this loop to publish 5 messages quickly. Watch the offset climb: 0 → 1 → 2 → 3 → 4. Each message gets the next slot in the partition log.
for i in 1 2 3 4 5; do curl -s -X POST http://localhost:8080/api/kafka/send \ -H "Authorization: Bearer $TOKEN" \ -H "Content-Type: application/json" \ -d "{\"action\": \"event-$i\"}" > /dev/null echo "Sent event-$i" done

Phase 4 — Retry & Dead Letter Topic

7
POST /api/kafka/send-fail — trigger the retry + DLT flow
The consumer throws RuntimeException for action="fail". Spring Kafka's @RetryableTopic catches this and routes the message through retry topics before parking it in the DLT. Watch ~15–20 seconds of log output.
curl -s -X POST http://localhost:8080/api/kafka/send-fail \ -H "Authorization: Bearer $TOKEN" | python3 -m json.tool
--- Attempt 1 (immediate) --- INFO KafkaMessageConsumer - Kafka received → topic=user-events offset=5 ERROR KafkaMessageConsumer - Simulated failure for action=fail --- Retry 1 (3s later, separate topic) --- INFO KafkaMessageConsumer - Kafka received → topic=user-events-0 offset=0 ERROR KafkaMessageConsumer - Simulated failure for action=fail --- Retry 2 (6s later) --- INFO KafkaMessageConsumer - Kafka received → topic=user-events-1 offset=0 ERROR KafkaMessageConsumer - Simulated failure for action=fail --- DLT (all retries exhausted) --- WARN KafkaMessageConsumer - DLT received (all retries exhausted) WARN KafkaMessageConsumer - DLT event: user=alice action=fail
Key insight: the main user-events topic is NOT blocked during retries. Other messages keep flowing through normally while this one bounces through retry topics. This is fundamentally different from RabbitMQ where a nacked message sits in the same queue.

Phase 5 — Advanced features

8
POST /api/kafka/send-batch — high-throughput batch publish
Send multiple events in one request. Kafka is designed for high-throughput sequential writes — this shows that strength. All events use the same key so they land in the same partition, preserving order.
curl -s -X POST http://localhost:8080/api/kafka/send-batch \ -H "Authorization: Bearer $TOKEN" \ -H "Content-Type: application/json" \ -d '[ {"action": "checkout"}, {"action": "payment"}, {"action": "confirm"}, {"action": "ship"}, {"action": "deliver"} ]' | python3 -m json.tool
Response: { "count": 5, "status": "sent", "key": "alice" } Logs show 5 consecutive messages, same partition, offsets in order: checkout → partition=1 offset=6 payment → partition=1 offset=7 confirm → partition=1 offset=8 ship → partition=1 offset=9 deliver → partition=1 offset=10
9
POST /api/kafka/send-partition — target a specific partition
Normally Kafka chooses the partition via key hash. Here you override that and send directly to partition 0. Watch the log confirm the partition matches what you requested.
curl -s -X POST http://localhost:8080/api/kafka/send-partition \ -H "Authorization: Bearer $TOKEN" \ -H "Content-Type: application/json" \ -d '{"action": "explicit-test", "partition": "0"}' | python3 -m json.tool
Log: INFO KafkaMessagePublisher - Published to explicit partition=0 offset=0 INFO KafkaMessageConsumer - Kafka received → partition=0 offset=0
10
Inspect topics inside the Kafka container
Use the Kafka CLI (inside the Docker container) to list all topics and describe the user-events topic. This shows partition assignments and leader info.
# list all topics docker exec kafka /opt/kafka/bin/kafka-topics.sh \ --bootstrap-server localhost:9092 --list # describe user-events topic (partitions, leaders, replicas) docker exec kafka /opt/kafka/bin/kafka-topics.sh \ --bootstrap-server localhost:9092 \ --describe --topic user-events # show consumer group offsets (which offset each partition is at) docker exec kafka /opt/kafka/bin/kafka-consumer-groups.sh \ --bootstrap-server localhost:9092 \ --describe --group demo-group
Expected output (kafka-topics --describe): Topic: user-events PartitionCount: 3 ReplicationFactor: 1 Partition: 0 Leader: 1 Replicas: 1 Partition: 1 Leader: 1 Replicas: 1 Partition: 2 Leader: 1 Replicas: 1 Expected output (consumer-groups --describe): GROUP TOPIC PARTITION CURRENT-OFFSET LOG-END-OFFSET LAG demo-group user-events 0 0 0 0 demo-group user-events 1 11 11 0 demo-group user-events 2 0 0 0
11
Replay messages from the beginning (unique to Kafka)
Unlike RabbitMQ, Kafka keeps messages after they're consumed. Reset the consumer group offset to replay all events from the start. This is impossible in RabbitMQ without explicit DLQ management.
# 1. Stop the Spring app first (Ctrl+C), then reset offsets: docker exec kafka /opt/kafka/bin/kafka-consumer-groups.sh \ --bootstrap-server localhost:9092 \ --group demo-group \ --topic user-events \ --reset-offsets --to-earliest --execute # 2. Restart the app — it will re-consume ALL messages from offset 0: mvn spring-boot:run
When to use this: replaying events for a new consumer feature, rebuilding a read model, or debugging by re-processing historical data.

Core concepts — what each one means in this demo

Topic

A named, ordered, persistent log. All messages you publish go into a topic. In this demo: user-events.

Think of it as an append-only file on disk, not a temporary queue. Messages stay even after they're consumed — retention policy decides when they're deleted (default: 7 days).

Partition

Topics are split into partitions for parallelism. user-events has 3 partitions.

Each partition is an independent ordered log. Messages within a partition are always consumed in order. Across partitions, order is not guaranteed.

Offset

The position of a message within a partition. Starts at 0, increments by 1 per message.

Consumers remember their offset. If a consumer crashes and restarts, it picks up exactly where it left off. This is Kafka's "at-least-once" delivery guarantee.

Consumer Group

A group of consumers that share the work. group-id: demo-group.

Kafka assigns each partition to exactly one consumer in the group. With 3 partitions and 1 consumer instance, one consumer handles all 3. Add a second instance and Kafka rebalances — each gets some partitions.

Message Key

An optional string sent with each message. In this demo, the key is the username.

Kafka uses hash(key) % partitionCount to decide the partition. Same key = same partition = messages for the same user are always in order. Without a key, round-robin is used.

Dead Letter Topic (DLT)

Where messages go when all retries fail. In this demo: user-events.DLT.

Spring Kafka's @RetryableTopic automatically creates intermediate retry topics and routes to the DLT after the last attempt. The @DltHandler method handles them.

Why does order matter? If you're modelling user actions — login → checkout → payment → ship — they must be processed in order. With key=username, all events for "alice" go to the same partition and are consumed in the same order they were published. This ordering guarantee is per-partition, per-consumer-group.

@RetryableTopic — how the retry chain works

1
user-events — attempt 1 (immediate)
Consumer reads the message. If it throws, Spring intercepts the exception.
2
user-events-0 — retry 1 (after 3 s)
The original message is forwarded to a retry topic. The main topic is free to process other messages immediately.
3
user-events-1 — retry 2 (after 6 s)
Still failing. Forwarded again to the second retry topic with a longer back-off.
4
user-events.DLT — final resting place
All attempts exhausted. The @DltHandler method receives the message. In production you'd persist it to a database or send an alert.

Same project, two brokers — spot the difference

Concept Kafka (this demo) RabbitMQ (this demo)
Model Distributed commit log — append-only, partitioned Message broker — exchange routes to queues
Message routing Producer selects topic + key → Kafka picks partition via hash Producer sends to exchange with routing key → exchange routes to queue
After consumption Message stays in the partition log until TTL expires. Can replay. Message deleted after ack. Replay requires DLQ intervention.
Ordering Guaranteed per partition. Same key → same partition → same order. Guaranteed within a single queue if there is one consumer.
Dead letters @RetryableTopic auto-creates retry topics + DLT. Main topic never blocked. DLX/DLK configured on queue. Nacked messages re-sit in same queue during retry (blocks ordering).
Consumer group One partition → one consumer per group. Add consumers = horizontal scale. Competing consumers on a queue. Multiple consumers share the queue.
Throughput Very high (sequential disk writes, batching, compression) High but lower than Kafka; message ack per-message adds overhead
Use case sweet spot Event streaming, audit logs, activity feeds, data pipelines Task queues, RPC, work distribution, transient jobs
Spring annotation @KafkaListener / @RetryableTopic @RabbitListener + retry interceptor
Send API KafkaTemplate.send(topic, key, value) → CompletableFuture RabbitTemplate.convertAndSend(exchange, routingKey, obj)
Config concept bootstrap-servers → topics → partitions → consumer-group host → exchange → binding → queue → dead-letter-exchange
Management UI No built-in UI. Use kafka-topics.sh CLI or Kafka UI (separate tool) Built-in web UI at :15672 (RabbitMQ Management plugin)

Side-by-side code comparison

Kafka — publish

// key = username → same user // always → same partition kafkaTemplate.send( topic, username, // key event );

RabbitMQ — publish

// routing key selects which // queue receives the message rabbitTemplate.convertAndSend( exchange, routingKey, event );

Kafka — consume

@RetryableTopic(attempts = "3") @KafkaListener(topics = "user-events") void handle( ConsumerRecord<String, UserEvent> r ) { // r.partition(), r.offset() // r.key(), r.value() }

RabbitMQ — consume

// retry configured in application.yml // max-attempts: 5, interval: 5000ms @RabbitListener( queues = "user.events.queue" ) void handle(UserEvent event) { // no offset, no partition }
When to choose Kafka over RabbitMQ:
• You need to replay past events (audit trail, new service bootstrap, bug investigation)
• You need strict per-user or per-entity ordering across many consumers
• You need very high throughput (millions of events/sec)
• Multiple independent services must consume the same events independently (each group gets its own offset)

When to stick with RabbitMQ:
• Simple task queue — do this job once, discard after done
• RPC-style request/reply patterns
• Rich routing (topic exchange, headers exchange) with complex binding rules
• You want a built-in management UI without extra tooling

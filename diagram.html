Login & Register Process Diagrams — based on actual source code

Browser
User's web browser
🍃
Spring Boot
:8080 (browserFilterChain)
🔑
Keycloak
:8180 realm: demo-realm
1
Browser → Spring Boot
GET http://52.65.212.193:8080/
User visits any protected page. No session cookie → Spring sees unauthenticated request.
2
Spring Boot → Browser
302 Redirect → /oauth2/authorization/keycloak
browserFilterChain (Order 2) detects no authentication. Configured .loginPage("/oauth2/authorization/keycloak") skips Spring's own login page and goes straight to Keycloak.
3
Browser → Spring Boot
GET /oauth2/authorization/keycloak
Spring's OAuth2 client builds the authorization URL using application.yml settings.
4
Spring Boot → Browser
302 Redirect → Keycloak Authorization Endpoint
GET :8180/realms/demo-realm/protocol/openid-connect/auth ?response_type=code &client_id=spring-demo-client &redirect_uri=http://52.65.212.193:8080/login/oauth2/code/keycloak &scope=openid profile email roles &state=random-csrf-token
5
Browser → Keycloak
GET Keycloak Login Page
Browser follows the redirect. Keycloak renders its own login form (username + password). Spring Boot is not involved at this point.
6
User → Keycloak
POST Keycloak Login Form Submit
User types username & password and submits. Keycloak validates credentials against its user store in demo-realm. Spring Boot never sees the password.
username=alice&password=secret&credentialId=
7
Keycloak → Browser
302 Redirect → Spring Boot Callback URL
GET http://52.65.212.193:8080/login/oauth2/code/keycloak ?code=AUTH_CODE_abc123 &state=random-csrf-token
Keycloak sends a short-lived authorization code (not the token yet).
8
Browser → Spring Boot
GET /login/oauth2/code/keycloak?code=AUTH_CODE
Spring receives the authorization code. Validates the state to prevent CSRF. Then performs a back-channel (server-to-server) token exchange.
9
Spring Boot → Keycloak (back-channel)
POST :8180/…/token
POST :8180/realms/demo-realm/protocol/openid-connect/token grant_type=authorization_code code=AUTH_CODE_abc123 redirect_uri=http://52.65.212.193:8080/login/oauth2/code/keycloak client_id=spring-demo-client client_secret=super-secret-client-key
10
Keycloak → Spring Boot (back-channel)
200 OK — Token Response
{ "access_token": "eyJhbG...", // used for API calls "id_token": "eyJhbG...", // OIDC: user identity "refresh_token": "eyJhbG...", // get new access token "expires_in": 300, "token_type": "Bearer" }
11
Spring Boot (internal)
Creates OidcUser session from ID token
Spring parses the id_token claims: preferred_username, email, given_name, family_name. Stores the user in the HTTP session. Scope openid is what triggered OIDC mode.
ID Token claims used by HomeController: user.getPreferredUsername() → "alice" user.getEmail() → "alice@example.com" user.getGivenName() → "Alice" user.getAuthorities() → [ROLE_USER, SCOPE_openid, ...]
12
Spring Boot → Browser
302 Redirect → / (defaultSuccessUrl)
Configured in SecurityConfig: .defaultSuccessUrl("/", true). Sets a session cookie on the browser.
13
Browser → Spring Boot
GET / (with session cookie)
Browser follows redirect, sends session cookie. Spring looks up the OidcUser from the session.
14
Spring Boot → Browser
200 OK — home.html (Thymeleaf)
HomeController.home() injects username, email, fullName, and roles into the Thymeleaf model. Renders templates/home.html.
model: username="alice" email="alice@example.com" fullName="Alice Smith" roles=["USER", "openid", "profile", "email"]
Key design choices in the code:
• No Spring login page — .loginPage("/oauth2/authorization/keycloak") in SecurityConfig sends users directly to Keycloak.
• Session-based — the browser chain uses HTTP sessions (not stateless). The API chain (/api/**) is stateless JWT.
• Two filter chains — @Order(1) apiFilterChain handles /api/**; @Order(2) browserFilterChain handles everything else.
🔌
API Client
curl / Postman / mobile app
🍃
Spring Boot
:8080 (apiFilterChain, STATELESS)
🔑
Keycloak
:8180 realm: demo-realm
1
API Client → Keycloak (direct — Spring not involved)
POST :8180/realms/demo-realm/protocol/openid-connect/token
Client sends credentials directly to Keycloak. Spring Boot never sees the password. Uses password grant (Resource Owner Password Credentials).
grant_type=password client_id=spring-demo-client client_secret=super-secret-client-key username=alice password=secret
2
Keycloak → API Client
200 OK — JWT Access Token
Keycloak checks credentials, looks up the user's roles in demo-realm, signs a JWT with its RSA private key.
{ "access_token": "eyJhbGciOiJSUzI1NiIs...", "expires_in": 300, "token_type": "Bearer" } JWT Payload contains: realm_access.roles: ["USER", "default-roles-demo-realm"] resource_access.spring-demo-client.roles: [...] preferred_username: "alice" iss: "http://52.65.212.193:8180/realms/demo-realm"
3
API Client → Spring Boot
GET :8080/api/user/profile
Client attaches the JWT in every request. Spring's apiFilterChain intercepts at /api/**.
Authorization: Bearer eyJhbGciOiJSUzI1NiIs...
4
Spring Boot → Keycloak (once at startup / on key rotation)
GET :8180/realms/demo-realm/protocol/openid-connect/certs
Spring fetches Keycloak's public RSA keys (JWK Set) on first request. Keys are cached — no round-trip to Keycloak per API call. Configured via jwk-set-uri in application.yml.
Response: JWK Set (Keycloak's public key) { "keys": [{ "kty":"RSA","kid":"abc123","n":"...","e":"AQAB" }] }
5
Spring Boot (local validation — no Keycloak call)
Validates JWT locally
Spring verifies: ① RSA signature (using cached public key), ② exp not expired, ③ iss matches issuer-uri. Stateless — no session, no DB lookup.
Checks: signature → valid (matches Keycloak public key) exp → not expired iss → "http://52.65.212.193:8180/realms/demo-realm" ✓
6
Spring Boot (KeycloakJwtRoleConverter)
Extracts roles → Spring Security authorities
KeycloakJwtRoleConverter reads realm_access.roles and resource_access.spring-demo-client.roles from the JWT. Maps each role to a ROLE_-prefixed Spring authority (uppercased).
realm_access.roles: ["USER"] → SimpleGrantedAuthority("ROLE_USER") resource_access["spring-demo-client"].roles: ["admin"] → SimpleGrantedAuthority("ROLE_ADMIN")
7
Spring Boot (SecurityConfig authorization check)
Access control check
SecurityConfig rules applied to /api/user/profile: requires ROLE_USER. Since the JWT has this role, access is granted and the controller runs.
SecurityConfig (apiFilterChain): /api/public/** → permitAll() /api/user/** → hasRole("USER") ← this request passes ✓ /api/admin/** → hasRole("ADMIN")
8
Spring Boot → API Client
200 OK — UserInfoDto JSON
UserController.getProfile() reads claims directly from the Jwt object (no DB call).
{ "username": "alice", "email": "alice@example.com", "subject": "user-uuid-from-keycloak", "roles": ["ROLE_USER"], "issuer": "http://52.65.212.193:8180/realms/demo-realm" }
Error responses:
• 401 Unauthorized — missing token, expired token, or invalid signature.
• 403 Forbidden — valid token but wrong role (e.g., USER trying to access /api/admin/**).
• No Keycloak call per request — JWT validation is fully local (signature check + expiry). Only the public key fetch hits Keycloak, and it's cached.

🔑
Keycloak
:8180 handles all registration
Registration is handled entirely by Keycloak. This Spring Boot app has no custom registration controller or form. When a user needs to register, they do so on Keycloak's built-in registration page. The Spring app only needs to enable User Registration in the Keycloak realm settings.
1
Browser → Spring Boot
GET http://52.65.212.193:8080/
New user visits the app. No session → Spring redirects through the same OAuth2 flow as login.
2
Spring Boot → Browser
302 → Keycloak Login Page
Same redirect chain as login: /oauth2/authorization/keycloak → Keycloak authorization endpoint.
3
Browser → Keycloak
Keycloak renders Login Page with "Register" link
Keycloak's login page shows a "Register" link if User Registration is enabled in Realm Settings → Login → User registration: ON.
4
New User → Keycloak
GET Keycloak Registration Page
User clicks "Register". Keycloak shows its own registration form.
GET :8180/realms/demo-realm/protocol/openid-connect/registrations ?client_id=spring-demo-client &redirect_uri=http://52.65.212.193:8080/login/oauth2/code/keycloak &response_type=code &scope=openid profile email roles
5
New User → Keycloak
POST Keycloak Registration Form Submit
User fills in and submits the registration form. Keycloak validates and creates the user account.
firstName=Alice lastName=Smith email=alice@example.com username=alice password=newpassword password-confirm=newpassword
6
Keycloak (internal)
Creates user account in demo-realm
Keycloak creates the user in its database, hashes the password, and assigns default realm roles (e.g., default-roles-demo-realm). Does NOT automatically assign USER or ADMIN — those must be assigned manually in Keycloak Admin UI or via Admin REST API.
Keycloak Admin UI → Users → alice → Role Mapping Assign realm role: USER ← must be done manually
7
Keycloak → Browser
302 → Spring Boot Callback (with authorization code)
After successful registration, Keycloak automatically logs the user in and sends the authorization code back to Spring Boot — same as step 7 in the login flow.
GET http://52.65.212.193:8080/login/oauth2/code/keycloak ?code=NEW_AUTH_CODE &state=random-csrf-token
8
Spring Boot → Browser
200 OK — home.html (logged in)
Spring Boot completes the token exchange (steps 9–13 from the login flow) and renders the home page. New user is now authenticated.
Important: After registration, Keycloak assigns only default realm roles. The new user will NOT have ROLE_USER or ROLE_ADMIN unless:
• You configure a Default Role in Keycloak (Realm Settings → User Registration → Default Roles), or
• An admin manually assigns roles via Keycloak Admin UI, or
• You use a Keycloak event listener to auto-assign roles on registration via the Admin REST API.

Browser
Authenticated user
🍃
Spring Boot
:8080 (session invalidation)
🔑
Keycloak
:8180 (SSO session destroy)
1
User → Browser
Clicks "Sign Out" button in home.html
The home.html template renders: <a href="/logout" class="logout-btn">Sign Out</a>
2
Browser → Spring Boot
GET /logout
Browser sends the GET request with the session cookie. Spring Security intercepts this URL.
3
Spring Boot (internal)
Destroys Spring Security session
Configured in SecurityConfig: .invalidateHttpSession(true) + .clearAuthentication(true). Removes the OidcUser from the session. Invalidates the session cookie.
SecurityConfig logout config: .logout(logout -> logout .logoutSuccessUrl("http://52.65.212.193:8180/realms/demo-realm" + "/protocol/openid-connect/logout" + "?redirect_uri=http://52.65.212.193:8080/oauth2/authorization/keycloak") .invalidateHttpSession(true) .clearAuthentication(true) )
4
Spring Boot → Browser
302 Redirect → Keycloak Logout Endpoint
GET :8180/realms/demo-realm/protocol/openid-connect/logout ?redirect_uri=http://52.65.212.193:8080/oauth2/authorization/keycloak
5
Browser → Keycloak
GET Keycloak Logout Endpoint
Browser follows the redirect to Keycloak's logout URL.
6
Keycloak (internal)
Destroys Keycloak SSO session & invalidates tokens
Keycloak ends the SSO session for this user. Refresh tokens are revoked. Other apps using the same Keycloak SSO session may also be logged out (back-channel logout, if configured).
7
Keycloak → Browser
302 Redirect → redirect_uri
Keycloak redirects to the redirect_uri specified in the logout URL: http://52.65.212.193:8080/oauth2/authorization/keycloak. This triggers the OAuth2 authorization flow again.
8
Browser → Spring Boot
GET /oauth2/authorization/keycloak
Spring redirects browser back to Keycloak's login page. The user sees the login form again and the cycle is complete.
Note: The redirect_uri after logout points to /oauth2/authorization/keycloak (the authorization initiation URL), not the app homepage. This means the browser is immediately redirected back to the Keycloak login page — the user cannot access any page of the app without logging in again.

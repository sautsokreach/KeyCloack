Testing Kafka Endpoints with Postman

Step-by-step guide — Spring Boot + Keycloak JWT + Kafka  |  Base URL: http://localhost:8080

Step 1 — Get a JWT Token from Keycloak
1
Obtain an access token (Password Grant)
⌄

All /api/kafka/* endpoints require a Bearer JWT issued by Keycloak. Use the Token endpoint below.

POST — Keycloak Token URL
POST http://52.65.212.193:8180/realms/demo-realm/protocol/openid-connect/token

In Postman, go to the Body tab → select x-www-form-urlencoded and add:

KeyValue
grant_typepassword
client_idspring-demo-client
client_secretsuper-secret-client-key
usernameyour Keycloak username
passwordyour Keycloak password
scopeopenid
Example Response
{ "access_token": "eyJhbGci...", "token_type": "Bearer", "expires_in": 300 }
Tip Copy the access_token value. You will paste it as a Bearer token in every request below. Tokens expire after ~5 minutes — re-run this request to get a fresh one.
2
Add the token to every request
⌄

In each Postman request, go to the Authorization tab and set:

FieldValue
TypeBearer Token
Tokenpaste the access_token here

Alternatively, add the header manually:

Header KeyHeader Value
AuthorizationBearer <access_token>
Content-Typeapplication/json (for POST requests)
Step 3 — Kafka Endpoints
GET
/api/kafka/info  — Topic configuration
⌄

Returns the Kafka topic setup and concept explanations. Good first request to confirm auth works.

Request
GET http://localhost:8080/api/kafka/info Authorization: Bearer <token>
Expected Response — 200 OK
{ "topic": "user-events", "partitions": 3, "consumerGroup": "demo-group", "concept": { "key-routing": "same key → same partition → ordered delivery per user", "offset": "position in partition log — consumers track where they left off", "consumer-group": "each partition is consumed by exactly one member of the group", "dlt": "dead letter topic receives messages after all retries fail", "vs-rabbitmq": "Kafka retains messages after consumption (replay possible); RabbitMQ deletes after ack" } }
POST
/api/kafka/send  — Send a user event
⌄

Publishes one event to the user-events topic. The message key is the authenticated username — same user always routes to the same partition.

Request
POST http://localhost:8080/api/kafka/send Authorization: Bearer <token> Content-Type: application/json { "action": "login" }

The action field is optional — defaults to "manual-trigger" if omitted.

Expected Response — 200 OK
{ "status": "sent", "topic": "user-events", "key": "alice", "action": "login" }

POST
/api/kafka/send-fail  — Trigger retry + DLT flow
⌄

Publishes an event with action = "fail". The consumer deliberately throws a RuntimeException, causing Spring Kafka to retry 3 times then route the message to the dead-letter topic.

Request (no body needed)
POST http://localhost:8080/api/kafka/send-fail Authorization: Bearer <token>
Expected Response — 200 OK
{ "status": "sent", "note": "consumer will fail → watch retry topics → DLT" }
What to watch in the app logs You should see 3 retry attempts logged, followed by the message appearing on user-events.DLT and the handleDlt consumer logging it.

POST
/api/kafka/send-batch  — Send multiple events
⌄

Publishes a list of events in sequence. Demonstrates Kafka's strength for high-throughput writes.

Request
POST http://localhost:8080/api/kafka/send-batch Authorization: Bearer <token> Content-Type: application/json [ { "action": "login" }, { "action": "view-dashboard" }, { "action": "logout" } ]
Expected Response — 200 OK
{ "status": "sent", "count": 3, "key": "alice" }
Tip You can send as many items as you like in the array. Each item with no action field defaults to "batch".
POST
/api/kafka/send-partition  — Send to a specific partition
⌄

Publishes directly to a named partition (0, 1, or 2). Useful for testing partition-level ordering.

Request
POST http://localhost:8080/api/kafka/send-partition Authorization: Bearer <token> Content-Type: application/json { "action": "test", "partition": "1" }

Valid partition values: "0", "1", "2" (topic has 3 partitions).

Expected Response — 200 OK
{ "status": "sent", "partition": "1", "key": "alice" }
Common Errors
!
Error reference
⌄
StatusCauseFix
401 Unauthorized Missing or expired Bearer token Re-run Step 1 to get a fresh token, paste it in Authorization
403 Forbidden Token valid but user lacks required role Ensure your Keycloak user has the correct realm/client roles assigned
400 Bad Request Malformed JSON body or wrong Content-Type Set Content-Type to application/json and verify JSON syntax
500 Internal Server Error Kafka broker not running Start Kafka on localhost:9092 before testing
Connection refused Spring Boot app not running Start the app with mvn spring-boot:run
